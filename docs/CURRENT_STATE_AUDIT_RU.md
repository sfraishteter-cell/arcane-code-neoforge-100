# Аудит текущего состояния Arcane Code

Дата аудита: 18 июля 2026 года.

Ветка: `feat/arcane-code-1.0.0-grand-weave`.

Базовый коммит `main`: `1636c13` (`Fix NeoForge client-to-server packet sending`). Переданный ZIP проверен отдельно. Он не является точной копией актуального `main`: содержит дополнительные вспомогательные файлы загрузки и отличается от репозитория интерфейсами, текстурами и документацией. ZIP не накладывался поверх более нового кода; актуальный GitHub-репозиторий принят за источник истины.

## Краткий вывод

Проект уже содержит компилируемый прототип игрового мода, а не пустую заготовку. В коде есть серверный интерпретатор, 142 базовые руны, динамические руны реестров, предметы, экраны, временные барьеры и рецепты Create. Однако прежнее обозначение `1.0.0` и формулировки release candidate были преждевременными: автоматических тестов нет, клиентская игра не проверена, Create не проверен в реальном запуске, а сетевой и мировой контуры недостаточно защищены для публичного сервера.

После этапа A проект собирается через Gradle Wrapper как `1.0.0-dev`, содержимое JAR проверяется, а dedicated server без Create достигает состояния `Done` без `ERROR`, исключений и ошибок загрузки рецептов.

- NeoForge 21.1.238 и Minecraft 1.21.1 разрешаются Gradle.
- `gradlew.bat --no-daemon clean build verifyReleaseJar --stacktrace` завершается успешно.
- Компиляционных ошибок нет.
- Итоговый файл: `build/libs/Arcane-Code-NeoForge-1.21.1-1.0.0-dev.jar`.
- JAR содержит `neoforge.mods.toml`, основные серверные и клиентские классы, локализации, модели, текстуры и рецепты.
- Версия в обработанном `neoforge.mods.toml` равна `1.0.0-dev`; незаменённого шаблона нет.

### Dedicated server

- `runServer` запущен на Java 21 без Create.
- NeoForge загрузил Arcane Code как `1.0.0-dev`.
- Загружено 1300 рецептов.
- Сервер достиг `Done (5.351s)!`.
- Повторный запуск на существующем тестовом мире успешен.

### Реализованный код игрового контура

- Современная регистрация NeoForge payload через `PayloadRegistrar.playToServer`.
- Клиент отправляет payload через `PacketDistributor.sendToServer`; устаревшего `ClientPacketDistributor` нет.
- Сервер сам находит посох, проверяет известные ID рун, списывает ресурс и исполняет программу.
- Интерпретатор имеет предел 8192 операций, глубину 48 и предел цикла 256.
- Реализованы типизированные иоты, стек, арифметика, логика, векторы, списки, цитирование и циклы.
- Для всех 142 базовых ID существует ветвь исполнения; это проверено статическим сопоставлением `RuneLibrary` и `RuneEngine`.
- Есть код серверных эффектов, телепортации, частиц, блоков, предметов, сущностей и временных барьеров.
- Есть ручной Резонансный жернов, четыре посоха, три носителя программ и Кодекс.
- Есть два экрана: начертание и пособие; пособие содержит 12 глав, поиск по каталогу и просмотр рецептов.
- Все пять рецептов Create теперь защищены `neoforge:mod_loaded` и игнорируются при отсутствии Create.

## Что не реализовано или не доказано

### Тесты

- Unit-тестов нет: Gradle сообщает `compileTestJava NO-SOURCE` и `test NO-SOURCE`.
- GameTest-классов и тестовых структур нет; включено только пространство имён GameTest.
- Нет тестов стека, математики, циклов, лимитов, распознавания, сериализации, стоимости и динамических реестров.
- Нет автоматического клиентского playtest, подключения клиента к dedicated server и проверки GUI Scale.
- Нет сборки или runtime-теста с установленным Create.

### Архитектура VM и реестра рун

- `RuneEngine` — монолитный класс с большим `switch`; метаданные находятся отдельно в `RuneLibrary`, но исполнение не расширяется через реестр операций.
- Реестр рун не data-driven: определения создаются программно, а не загружаются из datapack-данных с codec и reload listener.
- Динамические формы зависят от состава реестров установленной сборки; стабильность форм между разными наборами модов не гарантирована.
- `Iota.ENTITY` хранит прямую ссылку на `Entity`; нет устойчивой сериализации через UUID и безопасного повторного разрешения.
- Программы хранятся строкой CSV в `CustomData`; отсутствуют версия формата, codec и миграция.
- Функции пользователя, именованные переменные, области видимости и полноценный отладчик отсутствуют.

### Сеть и безопасность

- Клиент распознаёт рисунок локально и отправляет серверу готовые ID рун. Сервер не распознаёт исходный геометрический путь повторно.
- Payload ограничен 30 000 байтами и 512 ID, но нет rate limit, cooldown, nonce и защиты от повторного исполнения.
- Неизвестные ID молча отбрасываются до исполнения; некорректный `action` не получает явного отказа.
- Нет серверной конфигурации прав на разрушение/установку блоков, урон, призыв сущностей и выдачу предметов.
- Нет проверок загруженности чанка и world border перед большинством мировых действий.
- Нет журналирования опасных действий и разрешений администратора.

### Производительность и конструкции

- Лимит операций считает руну один раз, но не учитывает фактическое число созданных частиц или блоков.
- Купол и клетка создаются синхронными вложенными циклами; нет бюджета на тик, очереди продолжения и квоты конструкций игрока.
- Нет spatial index, реестра владельцев, сохранения конструкций, восстановления после перезапуска и пакетной сетевой синхронизации.
- Барьер — временный блок с запланированным tick. Это работающий базовый механизм, но не полноценная система конструкций.

### Пособие и интерфейсы

- Кодекс выдаётся при входе, если предмета нет в инвентаре. Постоянного флага «уже выдан» нет, поэтому предмет можно выбросить и получить снова.
- Есть 12 глав вместо требуемых 30.
- Нет дерева глав, истории переходов, избранного, прочитанного, новых глав, фильтров, тренировочного режима и клавиши открытия.
- Большая часть текста интерфейса зашита русскими строками в Java и не использует локализацию.
- Размеры экранов жёстко заданы как 760×430 и 780×440; адаптивность для маленького окна и разных GUI Scale не доказана.
- Нет Ctrl+Z/Ctrl+Y, полноценного redo, keyboard navigation и автоматических UI-тестов.

### Create

- Create объявлен необязательной зависимостью в `neoforge.mods.toml`.
- Рецепты Milling, Crushing и Mixing существуют и условно загружаются только с Create.
- Java-интеграции, Ponder-сцен, Mechanical Arm/Redstone Link API и теста с реальным Create нет.
- Совместимость с конкретной сборкой Create 6.x не доказана.

### Конфигурация и эксплуатация

- Нет CLIENT/SERVER/COMMON-конфигураций Arcane Code.
- Нет команд `/arcanecode ...`.
- Нет профилирования, метрик, журналов стоимости и инструмента очистки зависших конструкций.
- Нет datagen-контура и проверки сгенерированных данных.

## Проблемы исходной сборки до этапа A

1. В репозитории отсутствовали `gradlew`, `gradlew.bat` и `gradle-wrapper.jar`.
2. GitHub Actions устанавливал глобальный Gradle вместо использования Wrapper.
3. Версия преждевременно была `1.0.0` в Gradle, TOML, Java и ресурсах.
4. CI ожидал стабильный JAR `1.0.0`.
5. Не было автоматической проверки JSON, локализаций, моделей, текстур и версии внутри JAR.
6. Рецепты Create не имели условия `mod_loaded`, поэтому загрузка без Create не была защищена на уровне данных.
7. Dedicated server не запускался в CI.

Исходный код при запуске через отдельно установленный Gradle 8.10.2 и JDK 21 компилировался; ошибок Java-компиляции не обнаружено. Главный дефект сборки был инфраструктурным — отсутствующий Wrapper и неверный CI-процесс.
